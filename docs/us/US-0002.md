# US-0002 — Migration SQLite schema inicial

**As** Usuário local,
**I want** que meus dados kanban tenham schema persistente desde o primeiro uso,
**so that** quadros, colunas e cartões possam ser gravados de forma confiável.

## Intent

### Acceptance

- [x] Migration `YYYYMMDDHHMMSS_initial_schema.sql` cria boards, columns, cards, tags, card_tags, checklist_items, app_preferences.
- [x] Índices e FKs conforme `docs/architecture/data-model.md`.
- [x] App aplica migrations automaticamente ao iniciar; segunda execução não falha nem duplica schema.

### Why

Persistência é requisito central do produto. Antes desta US, o app é só UI vazia. Depois dela, o arquivo SQLite existe com estrutura completa para commands CRUD nas US seguintes.

### Where

v1-S1, EPIC-01. Depende de US-0001 (projeto Tauri). Desbloqueia US-0003 (commands) e toda gravação de dados.

## Plan

### Approach

- Criar migration `src-tauri/migrations/20260702173000_initial_schema.sql` com DDL de `boards`, `columns`, `cards`, `tags`, `card_tags`, `checklist_items` e `app_preferences` copiado de `data-model.md`, para que US-0003+ invoquem schema completo sem migration adicional.
- Implementar `src-tauri/src/db/mod.rs` com pool SQLite apontando para `app_data_dir/kanban.db` e runner de migrations no hook `setup` de `lib.rs`, garantindo idempotência na segunda abertura.
- Expor helper interno `with_connection` para commands futuros reutilizarem — features React nunca tocam SQL direto (DRY de acesso a dados).
