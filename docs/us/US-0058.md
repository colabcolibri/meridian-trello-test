# US-0058 — Schema version_epics e API relacional

**As** Usuário local,
**I want** que épico e versão fiquem ligados no banco de dados,
**so that** eu não precise digitar ou manter listas JSON duplicadas que podem divergir.

## Intent

### Acceptance

- [ ] Migration `YYYYMMDDHHMMSS_version_epics.sql` cria tabela `version_epics(project_id, version_id, epic_id)` com PK composta e FKs para `versions` e `epics`.
- [ ] Migration copia pares válidos de `epics.version_ids_json` para `version_epics` sem perda quando JSON parseável.
- [ ] Command `set_epic_versions` persiste vínculos N:N substituindo linhas anteriores do épico na transação.
- [ ] Command `list_epics_for_version` retorna épicos ligados à versão ordenados por id.
- [ ] Command `get_version_included` retorna estrutura derivada (épicos via junction + contagem opcional de US/sprints) — substitui leitura de `included_json` no write path.
- [ ] `create_epic` / `update_epic` deixam de exigir `version_ids_json`; serviço TS chama `set_epic_versions` após create/update.

### Why

Hoje épico grava versões em `version_ids_json` e versão aceita `included_json` digitado — duas representações da mesma relação sem integridade referencial. Este slice entrega a fundação SQLite e commands Tauri que todas as telas v5 reutilizam; sozinho não muda UI visível além de dados migrados corretamente.

### Where

v5-S1 / EPIC-19. Sem dependências — desbloqueia US-0059, US-0060, US-0063. Não altera hub nem wizards neste slice.

## Plan

### Approach

- Adicionar migration com timestamp `date +"%Y%m%d%H%M%S"` criando `version_epics` e script INSERT SELECT que parseia `version_ids_json` existente via JSON1 do SQLite, ignorando ids órfãos com log no stderr da migration.
- Implementar commands em `src-tauri/src/commands/agile/epics.rs` (ou módulo `relations.rs` se SRP exigir): `set_epic_versions`, `list_epics_for_version`, `get_version_included` — reutilizar padrão de transação e `project_id` composto dos commands existentes em `versions.rs` e `sprints.rs`.
- Estender `src/domain/agileTypes.ts` com tipo `VersionIncluded` (epics: {id, title}[], story_count?, sprint_count?) e `epicService`/`versionService` com métodos invoke correspondentes — uma fonte TS, sem duplicar parse JSON no frontend.
- Deprecar write de `version_ids_json` e `included_json`: commands de create/update versão/epic ignoram esses campos quando `set_epic_versions` ou Included derivado estiver ativo; manter colunas legadas read-only até v5 fechar.
